## §1 — First-assessment onboarding tour

**Why.** The activation funnel — signup → first finalized assessment —
is the single most important conversion metric for the free tier. The
tour is the primary instrument for moving that number.

**What landed.**
- `supabase/migrations/010_onboarding.sql` adds two columns:
  `has_completed_first_assessment` (default false) and
  `onboarding_dismissed_at`. Plus a partial index for "tour-eligible
  users" queries.
- `components/onboarding/FirstAssessmentTour.tsx` — five-step modal
  (Welcome / Calibrate meter / Set up zone / Capture readings /
  Generate report). Pure UI; doesn't navigate.
- `api/profile/mark-onboarded.ts` — POST endpoint with two actions:
  `'completed'` (sets `has_completed_first_assessment = true`) and
  `'dismissed'` (sets `onboarding_dismissed_at = NOW()`).
- `tests/components/onboarding-tour.test.tsx` — 13 tests covering the
  visibility predicate, the step progression, the spotlight at each
  step, dismiss behavior, and final-CTA behavior.

**Trigger logic.** A user sees the tour when:
1. `has_completed_first_assessment` is false, AND
2. `onboarding_dismissed_at` is null, AND
3. `created_at` is within 7 days of NOW().

The exported `shouldShowTour(profile)` helper encapsulates this. Any
caller (App.jsx, MobileApp.jsx) calls it before rendering the tour.

**Each step:**
| Step | Title | Spotlight |
|---|---|---|
| 1 | Welcome to AtmosFlow, {first name} | none |
| 2 | Tell us about your meter | meter calibration field |
| 3 | Add your first zone | zone "+" button |
| 4 | Tap through the questions | question-prompt screen |
| 5 | Finalize and download | finalize button |

Dismiss closes the modal and POSTs `action: 'dismissed'`. Final-step
CTA closes and POSTs `action: 'completed'`, then fires the
`onStartAssessment` callback (the parent navigates the user into the
assessment screen).

**Editing the tour.** The `TOUR_STEPS` constant in
`components/onboarding/FirstAssessmentTour.tsx` is the canonical copy.
Update the title/body/spotlight there; tests pin specific text.

## §2 — Public landing page

**Where.** `pages/index.tsx`. Mobile-first React component, dark-theme,
no external trackers. The early-access form is the only conversion
event tracked (POSTs to `/api/early-access`).

**Sections (in order).**
1. **Hero** — wordmark, headline, value-prop subhead, two CTAs (Start
   free → `/signup`, See sample report → `/sample-report.pdf`).
2. **Why AtmosFlow** — three cards: Faster, Defensible, Standards-driven.
3. **Sample report excerpt** — styled mockup of a per-parameter Results
   section, ending in a link to the full sample PDF.
4. **Pricing** — four-tier teaser: Free / Solo / Pro / Practice.
5. **Built for** — audience list + non-goals.
6. **Early Access** — prose + email form posting to `/api/early-access`.
7. **Footer** — firm, address, links to pricing/privacy/terms/contact.

**Editing copy.** The page is plain JSX; no CMS. Edit the file directly;
the test (`tests/pages/landing.test.tsx`) pins the section testids and
the CTA hrefs but does NOT pin marketing prose.

**Sample report PDF.** Generated by
`scripts/generate-sample-report-pdf.mjs` using `pdfkit`. The script
lives in the repo so the PDF can be regenerated with copy updates. To
re-render: `node scripts/generate-sample-report-pdf.mjs`.

**Performance constraints.**
- Mobile-first (390px iPhone Safari). Test with the dev tools device
  toolbar before committing layout changes.
- Lighthouse target ≥ 90 on mobile (LCP < 2.5s, no layout shift).
- No external trackers (no GA, no Segment, no Hotjar). Adding any
  external tracker requires explicit approval — we treat the early-
  access signup itself as the conversion event tracked in Supabase
  via `analytics_events`.

## §3 — Onboarding email sequence

**Templates.** `lib/email-sequences.ts` defines two sequences:

### Free tier (4 emails)
| ID | When | Cancels on |
|---|---|---|
| `free.welcome` | immediate (delay 0) | — |
| `free.sample` | signup + 2 days | first_assessment_completed |
| `free.activation` | signup + 5 days | first_assessment_completed |
| `free.success` | reactive (assessment_completed) | — |

### Paid tier (3 emails)
| ID | When |
|---|---|
| `paid.welcome` | immediate |
| `paid.tips` | signup + 3 days |
| `paid.feedback` | signup + 14 days |

**Editing email copy.** Each template is a `render(ctx) => { subject, body }`
function. Edit in `lib/email-sequences.ts`; tests pin the first-name
interpolation and the tier-specific subject lines but not body prose.

**Triggering.** `lib/email-triggers.ts` exposes:
- `enqueueSignupSequence(supabase, event)` — called after signup with
  `{ user_id, plan, signup_at }`. Inserts the timed templates as rows
  in `email_queue`.
- `onAssessmentCompleted(supabase, event)` — called after the user
  finalizes their first assessment. Cancels unsent
  `cancelOnMilestone='first_assessment_completed'` rows; for free tier,
  enqueues `free.success` immediately.
- `onSubscriptionCanceled(supabase, event)` — TODO. Currently a no-op.
  Wire up the "we're sorry to see you go" sequence when the copy is
  approved.

**Delivery.** `scripts/cron-email-queue-processor.ts` drains the queue
every 15 minutes via `api/cron-email-queue-processor.ts`. It uses
Resend (`RESEND_API_KEY`) and marks rows with `sent_at` and the Resend
message id. If `RESEND_API_KEY` is unset, rows are marked sent in
"dry-run" mode so the queue doesn't back up; this is logged as an
error in the result.

**Idempotency.** The cron's SELECT filter is
`scheduled_for <= NOW() AND sent_at IS NULL AND canceled_at IS NULL`.
The same row is never sent twice; concurrent cron runs are safe.

**Email queue schema.** `supabase/migrations/011_email_queue.sql`.
Partial index `idx_email_queue_scheduled` keeps the cron query cheap
as the table grows.

## §4 — Account settings (self-serve)

**Where.** `components/account/AccountSettings.tsx`. Surfaces:
- Plan + billing period + credits remaining + next renewal
- "Manage subscription" → `/api/customer-portal` (Group B)
- Profile fields (name, firm; email read-only)
- "Delete account" → two-step confirmation → `/api/delete-account` (Group A)

**Free-tier UX.** Free users see "Upgrade plan" instead of "Manage
subscription" — there's no Stripe customer to manage.

**Two-step delete.** Single-click never deletes. Click "Delete account"
→ first confirm → "Yes, continue" → second confirm → "Permanently
delete". Cancel at any stage exits the flow.

**Wiring into the SPA.** This is a self-contained component meant to be
mounted at `/account` or under Settings. The current
`SettingsScreen.jsx` in `src/components/` already has settings and is
where this gets imported when the SPA's settings screen is migrated.
That migration is a follow-up; the standalone component is acceptance-
green and routable independently.

## Activation metric — definition + where to find it

**Activation = signup → first finalized assessment**. Signal sources:
- Signup: `analytics_events` row with `event_type='signup_completed'`,
  or `profiles.created_at` (proxy if analytics drops).
- First finalized assessment: `analytics_events` row with
  `event_type='assessment_completed'`, OR
  `profiles.has_completed_first_assessment = true`.

**SQL example — % of signups in last 30 days who finalized a report:**

```sql
WITH cohort AS (
  SELECT id, created_at,
         has_completed_first_assessment AS activated
  FROM profiles
  WHERE created_at >= NOW() - INTERVAL '30 days'
)
SELECT
  COUNT(*) FILTER (WHERE activated)::float / NULLIF(COUNT(*), 0) AS activation_rate,
  COUNT(*) AS cohort_size
FROM cohort;
```

**Time-to-activation distribution** (median + p90):

```sql
WITH activation AS (
  SELECT p.id,
         p.created_at AS signed_up,
         a.created_at AS finalized
  FROM profiles p
  JOIN analytics_events a ON a.user_id = p.id AND a.event_type = 'assessment_completed'
  WHERE p.created_at >= NOW() - INTERVAL '90 days'
)
SELECT
  PERCENTILE_CONT(0.5)  WITHIN GROUP (ORDER BY EXTRACT(EPOCH FROM (finalized - signed_up))/3600) AS median_hours,
  PERCENTILE_CONT(0.9)  WITHIN GROUP (ORDER BY EXTRACT(EPOCH FROM (finalized - signed_up))/3600) AS p90_hours
FROM activation;
```

A healthy free-tier funnel: activation_rate > 25%, median < 48 hrs.
